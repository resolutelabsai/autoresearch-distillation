Independent Research

Abstract

We present a framework for training LLM agents via RL on any task improvement loop. An agent proposes modifications to a target file, experiments execute on remote machines, and metric outcomes drive policy updates through SDPO, GRPO, or other RL algorithms. Tasks are defined as YAML configs — specifying what file to edit, how to run experiments, and how to score results — making the system pluggable across domains: ML pretraining, GPU kernel optimization, prompt engineering, code speedups, and more.

On our flagship task — Karpathy's autoresearch benchmark — we train Qwen3-14B with SDPO to minimize validation loss on a GPT pretraining script. Over 495 training steps, SDPO converges to val_bpb ~1.025 and holds steady for 300+ steps, achieving a best of 1.023 (−3.1%). Our GRPO run — 38 steps with 128 rollouts/step (~94M tokens), after adding a KL penalty and batch normalization to stabilize training — plateaus at −1.8%. The SDPO-trained 14B model surpasses the Karpathy agent (−2.8%) while learning stable exploration patterns encoded in its weights.

Pluggable Task System

Every task-specific detail lives in a single YAML config: workspace, execution, scoring, prompts, feedback templates, and fleet slots. The agent loops, SSH runners, and training pipeline are fully generic — they read everything from the config at runtime.

Four Approaches (Autoresearch)

SDPO RL

The model generates rollouts — editing train.py via bash tool calls — and experiments execute on remote H100s. A reward signal is provided via hindsight context from a teacher model, flowing to the model's weights via policy gradient updates. SDPO does not need a successful demonstration for an advantage signal — it only needs the correct solution to be more likely under the teacher model compared to the student model, providing dense supervision even when all rollouts fail. Over 495 steps, SDPO entropy increases (0.28→0.61) — no mode collapse — while pg_loss steadily decreases (0.052→0.011), indicating continued learning.

SDPO is extremely sensitive to its training dynamics. We identified two failure modes that required active management. Entropy collapse kills exploration: the model locks into a narrow set of edits and stops trying new things. We addressed this with lower learning rate and entropy regularization. Teacher collapse kills the learning signal: when the EMA teacher becomes too similar to the student, pg_loss heads toward zero and the distributional mismatch that SDPO relies on vanishes. We fixed this by setting EMA τ = 0 (effectively freezing the teacher as the base model), after which we observed healthy entropy and pg_loss throughout training. The teacher-student gap must remain alive, or SDPO loses its reason to exist.

Environment and feedback design mattered as much as the algorithm itself. We added hash-based deduplication with a novelty reward: if a proposed diff was previously unseen, the feedback praised it; if it had been tried before, the model was told to try something new. We also built an experiment cache that cut iteration time from ~20 minutes to ~5 minutes per step — a 4× throughput improvement that, in a search-limited regime, mattered as much as any algorithmic change. Despite these measures, novelty rate (cache miss rate) fell to ~10% by step 100, suggesting the search process exhausts diversity early even when losses still look healthy.

GRPO RL

Group Relative Policy Optimization uses reward differences within a group as the learning signal. Our initial GRPO runs showed no return signal at all, so we made three changes: dropped the learning rate from 5e-6 to 1e-6 to reduce destructive updates in a sparse-reward setting, added a KL penalty (coefficient 1e-3) to prevent drift and collapse, and bumped the group size to 8 (128 rollouts/step vs SDPO’s 16) to increase the chance of seeing useful reward differences per step.

Over 38 steps (~94M tokens), our GRPO run reaches val_bpb = 1.037 (−1.8%) with entropy stable at ~0.30 but no continued improvement. A telling diagnostic: the KL term dominated pg_loss by 3–4 orders of magnitude, and gradient norms were very small. This suggests GRPO was operating more as “stay near base + better search” than as strong policy learning away from initialization.

Notably, once we added MCTS for structured proposal search, GRPO improved significantly and beat SDPO on best reward. This complicates the SDPO-vs-GRPO comparison: the improvement came primarily from better search scaffolding, not from richer GRPO learning dynamics. It suggests that on this task, proposal quality and search structure may matter more than the RL algorithm.

ICL Baseline In-Context

Same model, same tools, no weight updates. The model accumulates experiment feedback in its context window across 50 sequential turns: top results with full diffs, classified crash reports, and structured history. All learning happens through context conditioning. Qwen3-14B ICL achieves val_bpb = 1.038.

Single-turn Baseline No Feedback

N independent single-shot calls with no history and no feedback. Just “here’s train.py, improve it” repeated N times, taking the best result. This measures the entropy of the sampling distribution — what you get from pure random exploration at temperature 1.0. Qwen3-14B single-turn achieves val_bpb = 1.032 — beating its own ICL result of 1.038. The pattern is consistent: single-turn finds better peaks through variance, while ICL adds consistency without improving outcomes.

What the Model Finds

The dominant insight across both SDPO and ICL: trade model size for training throughput. Within the fixed 5-minute budget, a smaller model that completes more gradient steps outperforms a larger model that trains fewer steps. The best SDPO result combines ASPECT_RATIO 40, DEPTH 12, batch 64, and learning rate tuning — a compound edit that increases depth while reducing width and batch size, achieving 1.024 val_bpb.

Learned intuition through weight updates

The SDPO prompt contains zero experiment history — just the system prompt and train.py. Yet after training, the model generates chain-of-thought that references “previous attempts” and their outcomes. These aren’t memories from context; they’re learned intuitions encoded in the weights through SDPO, surfacing as recollections in chain-of-thought used to guide reasoning:

The model has never seen experiment feedback in its prompt. There is no “user” and no “previous attempt.” But through SDPO training on thousands of rollouts, the model’s weights have internalized which changes help, which crash, and why — all encoded as intuition, not context.

Best single-turn result (Claude Haiku 4.5, −4.4%)

The strongest single result across all methods came from Claude Haiku 4.5 on turn 42 of 50 independent single-shot attempts (no feedback, no history). The edit follows the same core insight — trade model size for throughput — but arrives at it in one shot:

--- a/train.py +++ b/train.py -TOTAL_BATCH_SIZE = 2**19 # ~524K tokens +TOTAL_BATCH_SIZE = 2**17 # ~131K tokens -EMBEDDING_LR = 0.6 +EMBEDDING_LR = 0.4 -MATRIX_LR = 0.04 +MATRIX_LR = 0.03 -WARMDOWN_RATIO = 0.5 +WARMDOWN_RATIO = 0.3 -FINAL_LR_FRAC = 0.0 +FINAL_LR_FRAC = 0.05 -DEPTH = 8 +DEPTH = 10 -DEVICE_BATCH_SIZE = 128 +DEVICE_BATCH_SIZE = 64

Result: val_bpb = 1.009 (−4.4%), completing 1,580 training steps vs 591 at default settings. Smaller batch (131K → 4× more steps), more depth (10 layers), halved device batch, lower learning rates, and a nonzero final LR fraction.

Likely a lucky outlier. Haiku only succeeded on 4 of 50 attempts with an average of 1.070. For comparison, Claude Opus 4.6 single-turn averaged 1.036 across 39 attempts (34/39 succeeded, best 1.015) — much more reliable but with a less extreme best case. The Haiku result is impressive but not reproducible; Opus shows what a stronger model achieves consistently.

Weights vs. Context

The core question: does RL training add value beyond what in-context learning provides?

Single-turn beats ICL. The single-turn baseline (1.032) outperforms ICL (1.038). Context conditioning adds reliability, not capability — and can even hurt when the model over-conditions on noisy feedback.

SDPO breaks out. Weight updates encode exploration patterns that persist across rollouts without consuming context. SDPO converges to ~1.025 and holds steady for 300+ steps with 16 rollouts/step (~137M tokens total). Our GRPO run, despite 128 rollouts/step (~94M tokens) and the addition of a KL penalty and batch-normalized advantages, plateaus at 1.037. The trained SDPO checkpoint (1.023) surpasses the Karpathy agent (0.970, −2.8%) from a 14B open-source model.

The gap between single-turn and ICL measures what context contributes (nothing — single-turn actually wins). The gap between both baselines and SDPO measures what weights contribute.

Why SDPO Outperforms GRPO

We ran four complementary analyses on the trained checkpoints to understand how SDPO (−3.1% in ~137M tokens, 16 rollouts/step) achieves better results than our GRPO run (−1.8% in ~94M tokens, 128 rollouts/step with KL penalty and batch-normalized advantages). The picture is consistent: SDPO makes targeted, low-rank weight changes that don’t alter internal activation geometry but meaningfully shift what the model predicts. Our GRPO checkpoint, even with 8× more rollouts per step, shows minimal functional divergence from the base model.

3. Activation Similarity

We measured per-token cosine similarity between base and fine-tuned model activations at layers 10, 20, 30, and 38 across 600 diverse sequences (C4, code, GSM8K, Wikipedia).

Result: cosine similarity > 0.999 everywhere for both models. Neither GRPO nor SDPO meaningfully changes the model’s internal activation geometry. The representational space is nearly identical to the base model. This means the weight changes operate at a finer granularity than activations can capture — they modify what the model outputs without changing how it represents inputs internally.

The Full Picture

Combining all four analyses: SDPO makes surgical, low-rank weight changes (SVD) that don’t alter activation geometry (cosine similarity > 0.999) but meaningfully shift predictions (1.6× more tokens changed, higher entropy) in a way that improves generative capabilities (GSM8K +1.21%, WikiText −0.1) without degrading knowledge (MMLU +0.02%). Our GRPO run — which required adding a KL penalty and batch normalization just to produce stable training — made changes too small and diffuse to show up in any of these analyses. The 8× rollout advantage (128 vs 16 per step) didn’t compensate for the constrained update magnitude.

Comparison to the Karpathy Agent

The original autoresearch experiments used an LLM-based agent on H100 GPUs, running 126 experiments over multiple cycles. The best result improved val_bpb by 2.8% (0.998 → 0.970). Key improvements: batch halving for more gradient steps (−0.012), adding depth (−0.004), embedding LR tuning (−0.003), and weight decay adjustments.

A key design difference: Karpathy’s setup accepts successful diffs into train.py, so each experiment builds on the last improvement — the file evolves over time. In our setup, the agent always receives the original unmodified train.py. This is deliberate: SDPO rollouts start from scratch each time (learning lives in the weights, not the file), so the baselines must do the same for a fair comparison. It means every improvement must be independently rediscovered, making our results harder to achieve but cleaner to interpret.

Despite this handicap, our SDPO-trained checkpoint evaluated with ICL (−3.1%) surpasses the original Karpathy agent (−2.8%), using a 14B open-source model trained with RL. The dominant insight is the same across all agents: throughput over parameters — trade model size for more training steps within the fixed budget. Both agents independently converge on this strategy.

Limitations & Future Work

Teacher model bias at small scale. SDPO’s teacher is the same model conditioned on hindsight feedback. At 14B, this creates a ceiling: the teacher can only “correct” the student toward solutions the base model could already express when given the right context. The teacher-student gap that drives SDPO’s learning signal is bounded by the base model’s own capabilities, which may explain why SDPO improved average behavior more than it discovered qualitatively new solutions. Larger base models or external teacher models (e.g., using a stronger model to generate feedback) could widen this ceiling.

Compound edits. Encouraging the model to combine multiple single edits that individually reduce validation loss remains a challenge. The best SDPO result is a compound edit, but these emerged rarely and late in training.

Rollout conflation. Although references to previous experiments in chain-of-thought serve as a means of improving reasoning quality, the model tends to conflate feedback from different rollouts. We suspect this is due to using a batch size greater than one with the same prompt and varying teacher hindsight context.

Search exhaustion. Novelty rate collapsed to ~10% by step 100 despite explicit novelty rewards and hash-based deduplication. The model cycles through a limited repertoire of edits rather than discovering genuinely new regions of the solution space. Improving proposal diversity — through better prompting, tool use, or external search — may matter more than further algorithmic improvements.

Built On

Autoresearch (Karpathy) — A single-file GPT pretraining script. The flagship benchmark: modify train.py to minimize val_bpb within a 5-minute training budget on a single H100.

SDPO (Hübotter et al., 2026) — Self-Distillation Policy Optimization. Converts tokenized environment feedback into a dense learning signal by treating the current model conditioned on feedback as a self-teacher, then distilling its corrected next-token predictions back into the policy. We extend it with agentic tool use and live environment rewards from real experiments.

VERL — RL training framework for LLMs. Our fork adds SDPO, agentic tool use, and multi-turn rollouts with configurable task environments.
